POSTDATA core ontology: Description back to ToC

The objective of this ontology is to provide the necessary knowledge for the poetic works and its manifestations representation.
It is important to note that some classes are alienated with the edm ontology.

The most relevant classes of POSTDATA core ontology are:

  • The pdcore:PoeticWork class, that represents an abstract concept of an artistic creation. In POSTDATA these creations must be in verse (poem, play, song). General data properties representing title, language, keyword, subtitles (if any), literary tradition, function (if any) or theme are included to instances of Poetic Work. For example: Os Lusíadas by Luis Vaz de Camões; La Commedia by Dante; the poem Ad te deus by an Anonymous author written in between the 7th and 9th centuries. This class is in turn aligned with the edm ontology as a edm:ProvidedCHO specialization as this class is used to represent Cultural Heritage objects that are aggregated with specific provider's data. This class is also aligned with a class of FRBRoo ontology: frbroo:F1 . The FRBRoo ontology was designed to reach a unified view of Cultural Heritage information with respecto to the standards and recommendations of libraries and museums. F1 represents an artistic or intellectual work, isolated from the artistic realisations of the work. The alignment will facilitate the retrieval of Poetic Works employing the mentioned external classes. The following diagram shows the data properties and object properties having class Poetic Work as range:
    The previous diagram shows the datatype properties (inside the class' box) and object properties (labeled arrows, possibly including inverse property name after "/") that are relevant for this class. As we can see, a Poetic Work is related to their creator or creators (CreatorRole) and several Redactions that are physical realisations of the work at hand. We can also specify a geographic location related to the provenance of the work and we could also find depicted characters (Role). In addition, we make use of several SKOS schemes to represent the possible values for the poetic type and genre of the work, as well as to express the educational level of the author. Finally, we can express different dates associated with the poem using imported classes from postdata-dates ontology.
  • The pdcore:Redaction class, that covers Poetic Work manifestations. It refers to a concrete version or instantiation of the Poetic Work. For example, one Poetic Work can have different editions; each edition represents one Redaction and they all refer to the same Poetic Work. As such, the textual elements as well as other aspects like the type of edition, witnesses used and not used, layout, explicit, incipit, or audience must be expressed using instances of Redaction. For example: The palaeographic edition of A solis ortu usque ad occidua preserved in Lat. 1154 (BNF) and its critical edition are two Redaction of the same Poetic Work. This class is a specialization of the class frbroo:F22, which is used inside frbroo to represent realisations of individual works. Find below the context diagram for the class Redaction:
    As we can see from the previous diagram, Redaction includes the data and object properties used to describe the textual elements of a work. To do so, we can make use of data properties like incipit or text. We can also make use of object properties to represent textual elements (TextUnit and OrderedTextUnitList) that can be further described employing POSTDATA Structural Elements ontology (stanzas, lines, words).
  • The pdcore:Ensamble class, that is an abstract work that contains more than one Poetic Work. An Ensemble might be conceived as a collection of works, organized by author, by book song, by book of poems; or as a work mainly written in prose that contains more than one poetic pieces (each one of them being its own Poetic Work). In case a prose book contains various poems, each poem constitutes one Poetic Work; at the same time, all poems contained in the prose book are part of the same Ensemble. We describe instances of this class through some properties like the total number of lines, poems and plays. Eg: Les fleurs du mal. This class is a frbroo:F17 specialization, which represents a group of works. The following diagram represents relevant properties available to describe instances of type Ensemble:
    As we can see, Ensembles have some Poetic Works that are part of it and can have a specific creator associated (CreatorRole). An Ensemble can also be related with provenance Place and can be referenced in a specific location of the text.

This ontology also models necessary classes to cover aspects regarding authorship, associated agents, places of origin and related events. We describe hereafter these classes including information about reused classes from other ontologies/vocabularies:

  • The pdcore:Person class and the pdcore:Organisation class, which are hierarchically dependent on the Agent class, as they share many functionalities. They are foaf:Person specialization and a foaf:Organisation specialization, respectively.

    We have a set of properties that can be used to express biographical data about a Person, including Places of birth and death. We also make use of several SKOS schemes to control the possible values of other personal data. As we can see, a Person may belong to an Organisation. It and may have an assocaited Role, or be directly associated to a Redaction as editor. In addition, we can express that a Person takes a special function in a Redaction (dedications, mentions, characters) or an Event in which is involved.

    Organisations are formed by Persons. They can be associated to a Place, or it could be and Agent involved in an Event. Organisations, like Persons, can also have an associated Role and be mentioned in a Redaction.

  • The pdcore:Place class, specialization of edm:Place, that represents a particular position, point, or area in space; a location.

    The previous figure shows that a Place can be described using latitude and longitude. It also may be mentioned in a Redaction and could be an element of an Event.

  • The pdcore:Event class, a specialization of edm:Event, that represents a thing that happens or takes place and that can be mentioned in a Redaction.

    Events may have some of its actor elements defined (Places and Agents). An Event can be mentioned in a Redaction.

  • In the ontology, there are also two important classes, the pdcore:Role class and pdcore:Creator Role class, a Role specialization .
    pdcore:Role is used to represent the role of the agents related to Poetic Work, its manifestations, its transmissions and other elements.
    pdcore:Creator Role is used for represent the function of a person or organization in the creation of a poetic work, redaction or other transmission element. This class also covers anonymous authorships and incorrect attributions.

    As we can see from the previous diagram, Roles are specific functions of Agents regarding a Poetic Work, Redaction or Ensemble. When the Role plays the function of creator, we make use of CreatorRole class, which includes an attribute (data property) to express that the author attribution has been agreed to be wrong. A SKOS scheme can be used to express other functions.

Also in the core there are properties of habitual use in different classes of the ontology because their semantics are the same for all of them. Many of these properties were defined through the properties reuse that belong to the following vocabularies dc and dcterms, and to schema , for example, dc:title, schema:url or others like pdcore:commentary

In addition, this ontology makes use of the classes and properties defined in postdata-dates ontology. The classes pddates:DateEntity and pddates:DateExpression can be used to describe temporal information about individuals of the previous classes. While the first class enables the aggregation of information such as the associated certainty, the second one and its subclasses are used to include the actual date values, possibly corresponding to an expressed interval.

The core ontology must be imported by the rest of the ontologies of the network of ontologies.

Generic RDF Example I

In this section we provide an example to show the basic elements of POSTDATA core ontology represented in RDF. Specifically, the example will show triplets for the poem (Poetic Work) "Suspiro y pruebo con la voz doliente" by Fernando de Herrera. The source of the text (Redaction) is provided by ADSO Corpus as part of a collection (Ensemble) entitled "Algunas obras de Fernando de Herrera" and can be accessed here in XML-TEI format.

Here we provide a simple translation of the mentioned XML file to RDF in turtle format respecting POSTDATA core ontology. After including the required prefixes, we describe an Ensemble and the Poetic Work that is part of the Ensemble. The following piece of data also expresses that the Poetic Work has a specific realsation (Redaction): 


@prefix cc: <http://creativecommons.org/ns#> .
@prefix dc: <http://purl.org/dc/elements/1.1/> .
@prefix olo: <http://purl.org/ontology/olo/core#> .
@prefix owl: <http://www.w3.org/2002/07/owl#> .
@prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix xml: <http://www.w3.org/XML/1998/namespace> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix skos: <http://www.w3.org/2004/02/skos/core#> .
@prefix vann: <http://purl.org/vocab/vann/> .
@prefix pdcore: <http://postdata.linhd.uned.es/ontology/postdata-core#> .
@prefix schema: <http://schema.org/> .
@prefix status: <http://www.w3.org/2003/06/sw-vocab-status/ns#> .
@prefix dcterms: <http://purl.org/dc/terms/> .
@prefix pddates: <http://postdata.linhd.uned.es/ontology/postdata-dates#> .
@prefix postdata: <http://postdata.linhd.uned.es/data/> .
@prefix partof: <http://www.ontologydesignpatterns.org/cp/owl/partof.owl> .
@prefix foaf: <http://xmlns.com/foaf/0.1>.
@prefix objectrole: <http://www.ontologydesignpatterns.org/cp/owl/objectrole.owl>.


postdata:Ensemble_1 a pdcore:Ensemble;
	pdcore:numberOfLines "14"^^xsd:integer;
	pdcore:numberOfPlays "0"^^xsd:integer;
	pdcore:numberOfPoems "91"^^xsd:integer;
	dc:title "Algunas obras de Fernando de Herrera";
	partof:hasPart postdata:Poetic_Work_1.


postdata:Poetic_Work_1 a pdcore:PoeticWork;
	pdcore:isSong "false"^^xsd:boolean;
	pdcore:isCalligram "false"^^xsd:boolean;
	pdcore:literaryTradition "Renaissance";
	pdcore:alternativeTitle "Suspiro y pruebo con la voz doliente";
	pdcore:keyword "Sufrimiento amoroso";
	pdcore:language "Spanish";
	pdcore:originalTitle "Soneto XL";
	pdcore:theme "El sufrimiento amoroso";
	pdcore:isRealisedThrough postdata:Redaction_1;
	pdcore:hasCreator postdata:Creator_Role_1.

The Redaction contains all the textual elements and information about the edition. In the next piece of the RDF example includes triplets to describe the Redaction and include a specific editor of it: 

postdata:Redaction_1 a pdcore:Redaction;
	pdcore:additionalFile "https://github.com/bncolorado/CorpusSonetosSigloDeOro/blob/master/FernandoDeHerrera/FernandoDeHerrera_161.xml"^^xsd:anyURI;
	pdcore:audience "Learned Spanish people interested in poetry";
	pdcore:explicit "así acabo la vida en el suspiro";
	pdcore:incipit "Suspiro y pruebo con la voz doliente";
	pdcore:numberOfWitnessesUsed "1"^^xsd:integer;
	pdcore:sampleText """Suspiro y pruebo con la voz doliente,
que en su dolor espire el alma mía,
crece el suspiro en vano y mi agonía,
y el mal renueva siempre su accidente""";
	pdcore:text """Suspiro y pruebo con la voz doliente,
que en su dolor espire el alma mía,
[..]
y como el cisne muere en dulce canto,
así acabo la vida en el suspiro.""";
	pdcore:typeOfRedaction "Digital edition";
	pdcore:unusedWitnesses "Ms. 10.159 of National Spanish Library (Cisnes del Betis)";
	pdcore:workNumber "40"^^xsd:integer;
	pdcore:language "Spanish";
	pdcore:originalTitle "Soneto XL";
	pdcore:hasEditor postdata:Person_2.

postdata:Person_2 a pdcore:Person;
	pdcore:name "Ramón";
	pdcore:forename "García González".

The last fragment of our example describes the creator of the poem: 

postdata:Creator_Role_1 a pdcore:CreatorRole;
	pdcore:isAnonymous "false"^^xsd:boolean;
	pdcore:isWrongAttribution "false"^^xsd:boolean;
	objectrole:isRoleOf postdata:Person_1.


postdata:Person_1 a pdcore:Person;
	pdcore:additionalName "El Divino"@es, "The Divine"@en;
	pdcore:biography """Although Herrera's family was only moderately wealthy, they [..] writer continued to work
up to his death in 1597 at the age of 63""";
	pdcore:ethnicity "Spanish";
	pdcore:forename "Fernando";
	pdcore:isFictional "false"^^xsd:boolean;
	pdcore:nameLink "de";
	pdcore:nationality "Spanish";
	pdcore:portrait "https://es.wikipedia.org/wiki/Fernando_de_Herrera#/media/Archivo:Francisco_pacheco-fernando_de_herrera.jpg"^^xsd:anyURI;
	pdcore:surname "Herrera";
	pdcore:name "Fernando".

Generic RDF Example II

Ensembles, Poetic Works and Redactions can be associated to the agents or roles that participated in the production of the mentioned items. In the following example, we define a creator and publisher roles for an ensemble that includes several poetic works. The following triplets express this idea: 

@prefix pdcore: <http://postdata.linhd.uned.es/ontology/postdata-core#> .
@prefix pddates: <http://postdata.linhd.uned.es/ontology/postdata-dates#> .
@prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix owl: <http://www.w3.org/2002/07/owl#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix dc: <http://purl.org/dc/elements/1.1/> .
@prefix postdata: <http://postdata.linhd.uned.es/data/> .
@prefix pdskos: <http://postdata.linhd.uned.es/kos/> .


postdata:Ensemble_1 a pdcore:Ensemble ;
	dc:title "Las cien mejores poesías de la lengua castellana" ;
	pdcore:hasCreator postdata:CreatorRole_1;
        pdcore:date postdata:date_Las-cien;
        pdcore:hasAgent postdata:Role_2;
        pdcore:hasProvenance ;
        pdcore:hasPart postdata:PW_Serranilla;
        pdcore:hasPart postdata:PW_A-la-Muerte.

postdata:CreatorRole_1 a pdcore:CreatorRole ;
	pdcore:isRoleOf postdata:Menendez-Pelayo;
        pdcore:isCreatorOf postdata:Ensemble_1;
        pdcore:certainty pdskos:definitely.

postdata:Role_2 a pdcore:Role;
        pdcore:certainty pdskos:definitely;
        pdcore:roleFunction pdskos:publisher;
        pdcore:isRoleOf postdata:libreria-VS.

As we can see, the ensemble was created by "Menendez Pelayo", but the publisher referes to another Agent. Here, we consider the publishing organisation as the publishing agent. Hence, we include further information about the Agents involved in the example, including the definition of author's birth date according to imported ontology classes from POSTDATA Dates Ontology: 

postdata:libreria-VS a pdcore:Organisation;
        pdcore:name "Libreria Victoriano Soriano";
        pdcore:hasRole postdata:Role_2.

postdata:Menendez-Pelayo a pdcore:Person ;
	pdcore:name "Marcelino " ;
        pdcore:birthDate postdata:BD-M-Menendez-Pelayo;
        pdcore:DeathDate postdata:DD-Menendez-Pelayo;
        pdcore:birthPlace postdata:Santander;
        owl:sameAs  .

postdata:Santander a pdcore:Place;
        pdcore:name "Santander";
        pdcore:typeOfPlace "Ciudad";
        owl:sameAs .

postdata:BD-M-Menendez-Pelayo a pddates:DateEntity;
	pddates:isExpressedAs postdata:Date1;
	pdcore:certainty pdskos:definitely.

postdata:Date1 a pddates:ExactDateExpression;
	pddates:dateExpressionAsdate "1856-11-3"^^xsd:date.

Generic RDF Example III

Redactions are realisations of Poetic Works. In this example, we show how to express relationships between different realisations of a Poetic Work. In this case, one of the Redactions is said to be a translation of another one. The complete RDF example is the following: 

@prefix pdcore: <http://postdata.linhd.uned.es/ontology/postdata-core#> .
@prefix pddates: <http://postdata.linhd.uned.es/ontology/postdata-dates#> .
@prefix rdf: <http://www.w3.org/1999/02/22-rdf-syntax-ns#> .
@prefix owl: <http://www.w3.org/2002/07/owl#> .
@prefix xsd: <http://www.w3.org/2001/XMLSchema#> .
@prefix rdfs: <http://www.w3.org/2000/01/rdf-schema#> .
@prefix dc: <http://purl.org/dc/elements/1.1/> .
@prefix postdata: <http://postdata.linhd.uned.es/data/> .
@prefix pdskos: <http://postdata.linhd.uned.es/kos/> .

postdata:Serranilla_1 a pdcore:PoeticWork;
	dc:title "Serranilla VI" ;
	pdcore:hasCreator postdata:Creator_Role_1 ;
	pdcore:isRealisedThrough postdata:Redaction_1;
        pdcore:language "es";
        pdcore:hasTranslation postdata:Redaction_Trans_2;
        pdcore:isPartOf postdata:Ensemble_1.

postdata:Marques-Santillana a pdcore:Person ;
	pdcore:name "Iñigo " ;
	pdcore:surname "López de Mendoza";
        pdcore:positionName "Marqués de Santillana".

postdata:Creator_Role_1 a pdcore:CreatorRole ;
	pdcore:isRoleOf postdata:Marques-Santillana .

postdata:Redaction_1 a pdcore:Redaction ;
	pdcore:originalTitle "Serranilla" ;
        pdcore:sampleText "Moza tan fermosa non ví en la frontera...";
        pdcore:realises postdata:Serranilla_1;
        pdcore:hasAgent postdata:Creator_Role_1.

postdata:Role_2 a pdcore:Role ;
	pdcore:isRoleOf postdata:Jose-J-Prats;
        pdcore:roleFunction pdskos:editor;
        pdcore:certainty pdskos:definitely.

postdata:Redaction_Trans_2 a pdcore:Redaction;
        pdcore:originalTitle "Serranilla" ;
        dc:Title "Serranilha VI";
        pdcore:language "Portugues";
            pdcore:sampleText "Moça tan fermosa non vi en la frontera..."
	pdcore:realises postdata:Serranilla_1;
	pdcore:hasAgent postdata:Role_2;
    pdcore:isTranslationOf postdata:Redaction_1.

As we can see, the same Poetic Work is realised trough two different redactions. One of the Redactions is a translation of the original one, which was translated to portuguese and so the title changes and there is the need to specify the original title of the translated Redaction.
